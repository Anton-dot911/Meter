# ТЗ №6 — Cost/Latency Tracker (внутрішній інструмент)

**Кодова назва:** Meter
**Тип:** бібліотека-обгортка + міні-дашборд, особисте користування
**Пріоритет:** Етап 0 (разом зі Scaffolder)
**Оцінка тривалості:** 2 дні

---

## 1. Призначення

Кожен LLM-виклик у всіх проєктах має бути видимим: скільки коштував, скільки тривав, чи впав, чи був retry. Без цього неможливо чесно писати метрики в README, оптимізувати промпти і ловити регресії вартості. Meter — тонкий шар логування навколо Anthropic SDK + один дашборд для всіх проєктів.

## 2. Функціональні вимоги

### MVP
- **F1. Обгортка SDK (TS і Python):** `meteredClient(client, {project, component})` — проксі навколо Anthropic-клієнта; перехоплює кожен виклик `messages.create` і після відповіді пише запис у лог. Нуль змін у бізнес-коді, крім ініціалізації.
- **F2. Запис логу:** `{ts, project, component, model, tokens_in, tokens_out, cost_usd, latency_ms, status[ok|error|retried], error_type?, request_id, trace_id?}`. Вартість рахується з таблиці цін моделей у конфігу (ціни звіряти з docs.claude.com і оновлювати вручну — окремий config-файл `prices.json` з датою актуальності).
- **F3. Транспорт:** запис у Supabase-таблицю `llm_calls` (спільна для всіх проєктів, поле `project` розділяє). Fallback при недоступності — локальний JSONL-файл, без падіння основного виклику (fire-and-forget).
- **F4. Дашборд (одна сторінка):** фільтри проєкт/період; картки: сумарна вартість, кількість викликів, error rate, p50/p95 latency; графік вартості по днях; таблиця останніх викликів; топ найдорожчих компонентів.
- **F5. Бюджет-сигнал:** конфігурований місячний ліміт на проєкт; перевищення → помітний банер на дашборді (без блокування викликів у MVP).

### v1.1
- F6. Прив'язка до trace_id процесів (інтеграція з Conveyor: вартість по кроках).
- F7. Порівняння вартості до/після зміни промпта (annotation-мітки релізів).
- F8. Опційне жорстке відсікання при перевищенні бюджету.

### Out of scope
Підтримка інших провайдерів; алерти в месенджери; мультикористувацькість.

## 3. Технічні рішення

- TS-пакет і Python-пакет з ідентичною схемою запису; живуть у монорепо інструменту; підключаються у проєкти як git-залежність (npm через `github:`, pip через VCS-URL).
- Дашборд: React + Vite (Netlify), читає Supabase напряму (особистий інструмент, RLS на власний акаунт).
- Тести: unit на розрахунок вартості; інтеграційний — обгортка не змінює відповідь SDK і не кидає виняток при відмові транспорту.
